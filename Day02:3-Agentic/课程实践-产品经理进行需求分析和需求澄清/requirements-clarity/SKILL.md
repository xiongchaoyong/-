# 需求澄清技能

## 描述

通过100分评分系统，将模糊需求自动转换为可执行的PRD产品需求文档。

## 指令

当被调用时，检测模糊需求：

1. **模糊的功能请求**
   - 用户说：「添加登录功能」、「实现支付」、「创建仪表盘」
   - 缺失：如何实现？用什么技术？有什么约束？

2. **缺失的技术上下文**
   - 未提及技术栈
   - 未确定集成点
   - 未指定性能/安全约束

3. **不完整的规格说明**
   - 无验收标准
   - 无成功指标
   - 未考虑边界情况
   - 未提及错误处理

4. **模糊的范围**
   - 边界不清晰（「用户管理」- 具体是什么？）
   - 未区分MVP和未来增强功能
   - 缺失「不包括什么」

**不应激活的情况**：
- 提及具体文件路径（如 "auth.go:45"）
- 包含代码片段
- 引用现有函数/类
- 有明确复现步骤的Bug修复

## 核心原则

1. **系统性提问**
   - 提出有针对性的具体问题
   - 每次一轮只问一类问题（2-3个问题）
   - 基于之前的回答递进
   - 避免让用户感到压力

2. **质量驱动迭代**
   - 持续评估清晰度评分（0-100分）
   - 系统性识别差距
   - 迭代直到≥90分
   - 记录所有澄清轮次

3. **可执行输出**
   - 生成具体的规格说明
   - 包含可衡量的验收标准
   - 提供可执行的阶段计划
   - 支持直接开始开发

## 澄清流程

### 步骤1：初始需求分析

**输入**：用户的需求描述

**任务**：
1. 解析并理解核心需求
2. 生成功能名称（kebab-case格式）
3. 确定文档版本（默认`1.0`，除非用户另有指定）
4. 确保`./docs/prds/`目录存在用于PRD输出
5. 进行初始清晰度评估（0-100分）

**评估标准**：
```
功能清晰度：/30分
- 明确的输入/输出：10分
- 定义了用户交互：10分
- 说明了成功标准：10分

技术具体性：/25分
- 提及技术栈：8分
- 确定集成点：8分
- 指定约束条件：9分

实现完整性：/25分
- 考虑了边界情况：8分
- 提及错误处理：9分
- 指定数据验证：8分

业务上下文：/20分
- 问题陈述清晰：7分
- 确定了目标用户：7分
- 定义了成功指标：6分
```

**初始响应格式**：
```markdown
我理解你的需求。让我帮助你完善这个规格说明。

**当前清晰度评分**：X/100

**清晰的方面**：
- [列出清晰的要点]

**需要澄清**：
- [列出差距]

让我系统性地澄清这些要点...
```

### 步骤2：差距分析

从四个维度识别缺失信息：

**1. 功能范围**
- 核心功能是什么？
- 边界是什么？
- 什么不包括在内？
- 边界情况有哪些？

**2. 用户交互**
- 用户如何交互？
- 输入是什么？
- 输出是什么？
- 成功/失败场景是什么？

**3. 技术约束**
- 性能要求？
- 兼容性要求？
- 安全考虑？
- 可扩展性需求？

**4. 业务价值**
- 解决什么问题？
- 目标用户是谁？
- 成功指标是什么？
- 优先级是什么？

### 步骤3：交互式澄清

**提问策略**：
1. 从影响最大的差距开始
2. 每轮2-3个问题
3. 逐步建立上下文
4. 使用用户的语言
5. 必要时提供示例

**提问格式**：
```markdown
我需要澄清以下几点来完成需求文档：

1. **[类别]**：[具体问题]？
   - 例如：[如有帮助的示例]

2. **[类别]**：[具体问题]？

3. **[类别]**：[具体问题]？

请提供你的答案，我将继续完善PRD。
```

**用户每次响应后**：
1. 更新清晰度评分
2. 在工作PRD大纲中记录新信息
3. 识别剩余差距
4. 如果评分<90：继续下一轮问题
5. 如果评分≥90：进入PRD生成

**评分更新格式**：
```markdown
感谢你提供的补充信息！

**清晰度评分更新**：X/100 → Y/100

**新澄清的内容**：
- [总结新信息]

**剩余需澄清的要点**：
- [如果评分<90，列出剩余差距]

[如果评分<90：继续下一轮问题]
[如果评分≥90：「完美！现在我将生成完整的PRD文档...」]
```

### 步骤4：PRD生成

一旦清晰度评分≥90，生成完整的PRD。

**输出文件**：

1. **最终PRD**：`./docs/prds/{feature_name}-v{version}-prd.md`

使用`Write`工具创建或更新此文件。`{version}`来自PRD中记录的文档版本（默认`1.0`）。

## 行为准则

### 应该做
- 提有针对性的具体问题
- 基于之前的回答递进
- 提供示例引导用户
- 保持对话语气
- 在PRD中总结澄清轮次
- 使用清晰、专业的语言
- 生成具体的规格说明
- 保持澄清模式直到评分≥90

### 不应该做
- 一次问完所有问题
- 未经确认做假设
- 在90分之前生成PRD
- 跳过任何必需部分
- 使用模糊或抽象的语言
- 在用户未响应时继续
- 过早退出技能模式

## 成功标准

- 清晰度评分≥90/100
- 所有PRD部分内容完整
- 验收标准可检查（使用`- [ ]`格式）
- 执行阶段包含具体可执行任务
- 用户批准最终PRD
- 准备好移交给开发
