# 操作系统进程管理总结

## 一、进程的基本概念

进程是程序在计算机上的一次执行活动，是系统进行资源分配和调度的基本单位。

### 1.1 进程的三种基本状态

| 状态 | 说明 |
|------|------|
| **就绪（Ready）** | 进程已获得除 CPU 之外的所有所需资源，等待 CPU 调度 |
| **运行（Running）** | 进程已获得 CPU，正在执行 |
| **阻塞（Blocked）** | 进程因等待某事件（如 I/O 完成、信号量）而暂停执行 |

状态转换关系：

```
就绪 --[调度]--> 运行
运行 --[时间片用完]--> 就绪
运行 --[I/O 请求或等待事件]--> 阻塞
阻塞 --[I/O 完成或事件到达]--> 就绪
```

### 1.2 PCB（进程控制块）

PCB（Process Control Block）是操作系统为每个进程维护的核心数据结构，用于保存进程的全部信息。主要包含：

- **进程标识符（PID）**：唯一标识一个进程
- **程序计数器**：记录下一条要执行的指令地址
- **CPU 寄存器**：通用寄存器、栈指针、状态寄存器等
- **内存管理信息**：页表、段表、内存限制等
- **I/O 状态信息**：分配给进程的 I/O 设备列表、打开的文件列表
- **调度信息**：进程状态、优先级、调度队列指针等
- **记账信息**：CPU 使用时间、已执行时间、时间限制等

PCB 是进程存在的唯一标志，操作系统通过 PCB 来感知和管理进程。

## 二、进程调度算法

### 2.1 先来先服务（FCFS）

- **原理**：按进程到达就绪队列的先后顺序进行调度
- **特点**：非抢占式，实现简单
- **优点**：公平，易于实现
- **缺点**：平均等待时间较长，不利于短作业； convoy effect（护航效应），短作业被长作业阻塞

### 2.2 短作业优先（SJF）

- **原理**：选择估计运行时间最短的进程优先执行
- **特点**：可以是抢占式（SRTF，最短剩余时间优先）或非抢占式
- **优点**：具有最短的平均周转时间，理论上最优
- **缺点**：难以准确预估进程运行时间；可能导致长作业饥饿（长期得不到调度）

### 2.3 时间片轮转（RR）

- **原理**：将 CPU 时间划分为固定长度的时间片，按 FCFS 顺序轮流分配
- **特点**：抢占式，时间片大小是关键参数
- **优点**：响应时间短，适合分时系统；公平性好
- **缺点**：时间片太小时上下文切换开销大；时间片太大时退化为 FCFS

### 2.4 多级反馈队列

- **原理**：设置多个不同优先级的就绪队列，每个队列采用不同的调度算法（如高优先级队列用 RR 且时间片短，低优先级队列用 FCFS）
- **特点**：进程可在队列间动态迁移（用完时间片降级，优先级高的可抢占）
- **优点**：兼顾交互性和吞吐量；短作业快速响应，长作业也能得到服务
- **缺点**：参数配置复杂（队列数量、时间片大小、升降级策略）

**对比总结：**

| 算法 | 抢占性 | 公平性 | 平均周转时间 | 适用场景 |
|------|--------|--------|-------------|----------|
| FCFS | 非抢占 | 好 | 较差 | 批处理系统 |
| SJF | 可抢占 | 差（长作业可能饥饿） | 最优 | 批处理系统 |
| RR | 抢占 | 很好 | 一般 | 分时系统 |
| 多级反馈队列 | 抢占 | 好 | 较好 | 通用操作系统 |

## 三、线程

### 3.1 线程与进程的区别

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源拥有 | 独立地址空间，拥有代码段、数据段、堆 | 共享进程的代码段和数据段 |
| 栈 | 独立的栈 | 每个线程拥有**独立的栈** |
| 切换开销 | 大（涉及地址空间切换、TLB 刷新） | 小（共享地址空间） |
| 通信方式 | 进程间通信（IPC），开销大 | 直接读写共享数据，开销小 |
| 系统资源 | 创建/销毁开销大 | 创建/销毁开销小 |

### 3.2 线程的组成

- **线程 ID**：唯一标识
- **程序计数器**：记录执行位置
- **寄存器组**：线程上下文
- **栈**：保存局部变量和函数调用信息

线程自己不拥有系统资源，但与同属一个进程的其他线程共享该进程的资源。

### 3.3 线程模型

- **用户级线程**：线程管理由用户空间的线程库完成，内核不可见；切换快，但一个线程阻塞会导致整个进程阻塞
- **内核级线程**：由内核管理，阻塞互不影响，但切换开销较大
- **混合模型**：结合两者优点

## 四、进程同步

### 4.1 互斥锁（Mutex）

- 用于保护临界区，保证同一时刻只有一个线程能访问共享资源
- 提供 `lock()` 和 `unlock()` 操作
- 简单直接，但可能导致忙等待（spinlock）

### 4.2 信号量与 PV 操作

信号量是由 Dijkstra 提出的同步机制，核心是 **PV 操作**：

- **P 操作（wait / down）**：信号量值减 1；若结果小于 0，进程阻塞
- **V 操作（signal / up）**：信号量值加 1；若结果 <= 0，唤醒一个阻塞进程

**信号量分类：**

- **整型信号量**：只能通过 PV 操作访问的整型变量
- **记录型信号量**：除数值外还维护一个等待队列（解决忙等待问题）

**典型用法：**

- 二值信号量（值为 0 或 1）用于互斥
- 计数信号量用于资源管理（如有限缓冲区）

### 4.3 管程（Monitor）

- 一种高级同步结构，封装了共享变量、对变量的操作以及同步逻辑
- 管程保证**同一时刻只有一个进程在管程内活动**
- 引入条件变量（condition variable）及其 `wait()` 和 `signal()` 操作
- 比信号量更结构化，不易出错

### 4.4 三种同步机制的对比

| 机制 | 抽象层次 | 易用性 | 典型问题 |
|------|---------|--------|---------|
| 互斥锁 | 低 | 容易误用 | 死锁、死循环 |
| 信号量 | 中 | 需要谨慎配对 PV | 遗漏 V 操作导致死锁 |
| 管程 | 高 | 相对安全 | 条件变量使用不当 |

## 五、经典同步问题

### 5.1 生产者-消费者问题

**问题描述**：生产者向有限缓冲区放入数据，消费者从缓冲区取出数据。需要保证：

1. 缓冲区满时生产者不能继续放入
2. 缓冲区空时消费者不能继续取出
3. 对缓冲区的访问互斥

**解决方案**：需要三个信号量

- `mutex = 1`：互斥访问缓冲区
- `empty = N`：空闲缓冲区数量
- `full = 0`：已填充缓冲区数量

```
生产者：                   消费者：
  P(empty)                   P(full)
  P(mutex)                   P(mutex)
  放入数据                    取出数据
  V(mutex)                   V(mutex)
  V(full)                    V(empty)
```

### 5.2 读者-写者问题

**问题描述**：多个读者可同时读数据，但写者必须独占访问。

- **读者优先**：只要有读者在读，后来的读者可直接进入，写者可能饥饿
- **写者优先**：一旦有写者等待，新读者不允许进入，防止写者饥饿

**核心思路**：通过计数器记录读者数量，并用互斥锁保护计数器的更新。

### 5.3 哲学家进餐问题

**问题描述**：5 位哲学家围坐圆桌，每人左右各有一根筷子。哲学家思考、饥饿、进餐循环，进餐需要同时拿起左右两根筷子。

**解决方案要点**：

- 防止死锁：限制最多 4 人同时拿起筷子；或要求哲学家必须同时拿起两根筷子（要么不拿）；或使用奇偶编号哲学家不同的拿筷顺序
- 破坏循环等待条件可有效避免死锁

## 六、死锁

### 6.1 死锁的定义

多个进程因竞争资源而互相等待，导致所有进程都无法继续执行的状态。

### 6.2 死锁的四个必要条件

| 条件 | 说明 |
|------|------|
| **互斥（Mutual Exclusion）** | 资源一次只能被一个进程使用 |
| **持有并等待（Hold and Wait）** | 进程持有一个资源的同时还在等待其他进程持有的资源 |
| **不可剥夺（No Preemption）** | 资源只能由持有它的进程主动释放，不能被强制剥夺 |
| **循环等待（Circular Wait）** | 存在进程资源的循环等待链：P1 等 P2 的资源，P2 等 P3 的资源，...，Pn 等 P1 的资源 |

四个条件缺一不可，只要破坏任意一个即可避免死锁。

### 6.3 死锁的处理策略

| 策略 | 说明 |
|------|------|
| **预防** | 破坏死锁的四个必要条件之一（如一次性申请所有资源、资源编号顺序分配） |
| **避免** | 每次资源分配前检查系统是否处于安全状态 |
| **检测** | 允许死锁发生，定期检测并恢复（如资源分配图检测） |
| **恢复** | 终止进程或剥夺资源以解除死锁 |

### 6.4 银行家算法（避免死锁）

**核心思想**：系统在分配资源前，先计算分配后系统是否处于**安全状态**。如果是，才进行分配；否则进程必须等待。

**关键数据结构：**

- **Available**：每种资源的可用实例数
- **Max**：每个进程对每种资源的最大需求
- **Allocation**：每个进程已获得的资源数
- **Need**：每个进程还需要的资源数（Need = Max - Allocation）

**安全性检查算法**：

1. 模拟资源分配，找到一个 Need <= Available 的进程
2. 假设该进程完成并释放资源，更新 Available
3. 重复直到所有进程都能完成（安全状态）或找不到符合条件的进程（不安全状态）

**优点**：相比死锁预防，资源利用率更高
**缺点**：需要预先知道进程的最大资源需求；进程数量可能动态变化，实际应用受限
