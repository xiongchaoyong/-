# TCP 三次握手与四次挥手 & 拥塞控制算法 —— 用费曼学习法讲清楚

> **费曼学习法核心**：如果你不能用最简单的语言把一件事讲给一个完全不懂的人听，说明你自己还没真懂。下面我用"打电话"和"寄快递"的类比，把 TCP 那点事说明白。

---

## 一、三次握手（Three-Way Handshake）—— 为什么是三次？

### 先讲场景：两个人打电话

想象你（客户端）要给朋友（服务器）打电话，目的是**建立一条可靠的通信线路**。

- **第一次**：你拨号 -> 朋友听到铃声（你发出连接请求，SYN）
- **第二次**：朋友接起电话 -> "喂？我听到了，你说吧"（朋友确认收到，SYN+ACK）
- **第三次**：你回应 -> "好的，那我开始说了"（你确认收到朋友的确认，ACK）

到了这一步，双方都知道"对方能听到自己"，开始正常通话。这就是三次握手。

### 为什么两次不行？为什么四次多余？

**两次握手的问题：**

假设只有两次：

1. 你发送："喂？"（SYN）
2. 朋友回复："听到了，你说吧"（SYN+ACK）

此时朋友以为连接已建立，开始等你说话。但你的"喂"可能因为网络拥堵，在中间**滞留**了一段时间才到达朋友那里。实际上你已经放弃这次呼叫了。朋友却在傻等——这浪费了资源。

更危险的场景：一个**旧的、滞留的连接请求**突然到达服务器，服务器以为这是新连接，回复了确认，连接就建立了。但实际上客户端根本没想连，于是服务器干等，资源被白白占用。

**三次握手解决了这个问题：**

第三次的 ACK 确保"客户端确实在线上，并且确实想要建立这个连接"。如果客户端根本没发起连接，就不会发这个 ACK，服务器也不会浪费资源。

**四次为什么多余？**

三次已经让双方都确认了"你发我收、我发你收"的双向能力，多一次就是浪费往返时间。

### 用一句话回答

> 三次握手是为了**防止失效的连接请求到达服务器后，服务器误以为要建立连接而白白等待**。第一次是客户端问"在吗"，第二次是服务器答"在，你呢"，第三次是客户端答"我也在"——三次刚好互相确认了彼此都在线。

### 三次握手的细节（写给想记住底层的人）

| 步骤 | 发送方 -> 接收方 | 报文含义 | 状态变化 |
|------|----------------|---------|---------|
| 1 | Client -> Server | SYN=1, seq=x | Client 进入 SYN_SENT |
| 2 | Server -> Client | SYN=1, ACK=1, seq=y, ack=x+1 | Server 进入 SYN_RCVD |
| 3 | Client -> Server | ACK=1, seq=x+1, ack=y+1 | 双方进入 ESTABLISHED |

**关键记忆点：**
- SYN 是"我想建立连接"的标识
- ACK 是"我收到了你的消息"的标识
- seq（序列号）是"我这次发的是第几号包裹"
- ack（确认号）是"我已经收到了你发到第几号，请从这个号之后继续发"

## 二、四次挥手（Four-Way Wavehand）—— 为什么是四次？

### 先讲场景：两个人挂电话

继续用打电话的类比。现在你们聊完了，要挂断电话。

- **第一次**：你说"我说完了，挂了啊"（客户端发 FIN）
- **第二次**：朋友说"好，我知道了，你等一下我还有最后一句要说"（服务器回 ACK，但还没挂）
- **第三次**：朋友把最后一句说完"那我也说完了，挂了啊"（服务器发 FIN）
- **第四次**：你说"好，挂了"（客户端回 ACK，然后等待一会儿再真正关闭）

### 为什么是四次，而不是三次？

核心原因：**TCP 连接是全双工的**——相当于一条双向马路，两个方向的车流是独立的。

- 关闭连接需要**两个方向分别关闭**
- 你停止发送（第一次挥手），不代表对方也停止了发送（对方可能还有数据没传完）
- 所以你的"我要关了"（FIN）和对方的"我也要关了"（FIN）**不能合并**，需要各自发一次

**用类比理解：**

- 三次握手 = "双方同时确认对方在线"（相当于双向同时握手，一次交换就够了）
- 四次挥手 = "双方分别说再见"（你说再见后，对方可能还没说完，得等他讲完再跟我说再见）

**三次挥手不行吗？**

不行。如果服务器收到 FIN 后立即也发 FIN（合并 ACK 和 FIN 为一次），那就是三次挥手。但 TCP 不允许这样，因为服务器收到 FIN 时可能**还有数据没发完**，它必须先 ACK 表示"我知道你要关了"，等自己发完数据再 FIN。

### 四次挥手的细节

| 步骤 | 发送方 -> 接收方 | 含义 | 状态变化 |
|------|----------------|------|---------|
| 1 | Client -> Server | FIN=1, seq=u | Client 进入 FIN_WAIT_1 |
| 2 | Server -> Client | ACK=1, seq=v, ack=u+1 | Server 进入 CLOSE_WAIT |
| 3 | Server -> Client | FIN=1, ACK=1, seq=w, ack=u+1 | Server 进入 LAST_ACK |
| 4 | Client -> Server | ACK=1, seq=u+1, ack=w+1 | Client 进入 TIME_WAIT（2MSL 后关闭） |

**关键记忆点：**

- 第 2 步和第 3 步**不能合并**，因为中间可能隔了一段时间（服务器要处理剩余数据）
- 第 4 步后客户端要等 **2MSL**（两倍最大报文段寿命）才真正关闭，是为了防止最后一个 ACK 丢失，导致服务器重发 FIN

### 一句话总结三次握手 vs 四次挥手

> **三次握手**是两个人同时确认"你能听到我、我能听到你"，一次双向确认就够了；**四次挥手**是两个人分别说"我说完了"和"我也说完了"，因为你说完了不代表我也说完了，需要各说各的再见。

## 三、TCP 拥塞控制四算法 —— 像是一个快递公司的运输策略

### 先建立直觉

想象你是一家快递公司（TCP 发送方），要从北京往上海运货。马路（网络）的运输能力是有限的，你不知道路上现在堵不堵。

你的目标：**尽可能快地把货送到，但又不能把路堵死。** 怎么办？

- 刚开始不知道路况 -> 先发少量货试探 -> **慢开始**
- 感觉路况不错 -> 慢慢加速到极限 -> **拥塞避免**
- 突然发现堵车了（丢包了）-> 赶紧减速 -> **快重传 + 快恢复**

### 算法一：慢开始（Slow Start）

**直觉：**

你刚开始送货，不知道路况。如果一上来就发 100 辆货车，可能直接把路堵死。所以你**先发 1 辆**，看看能不能顺利到达。如果到了，下次发 2 辆；2 辆都到了，下次发 4 辆。**指数增长。**

**关键点：**

- 初始拥塞窗口（cwnd）= 1 个报文段（MSS）
- 每收到一个 ACK，cwnd 翻倍（实际是每轮 RTT 翻倍）
- 增长速度是 **指数级**的：1 -> 2 -> 4 -> 8 -> 16 ...
- 直到达到 **慢开始门限（ssthresh）**，然后切换为拥塞避免

**为什么叫"慢开始"？**

其实增长速度是"指数级"的，一点都不慢。之所以叫"慢"，是相对于一开始就发大量数据而言的 —— 它**从 1 开始慢慢试探**，而不是一上来就塞满网络。

### 算法二：拥塞避免（Congestion Avoidance）

**直觉：**

当窗口大小增长到 ssthresh（比如 16），你觉得"路况大概摸清楚了"，不能再猛加速了，否则会撞车。于是改为**线性增长**：每经过一个 RTT，窗口只增加 1（而不是翻倍）。

**关键点：**

- cwnd >= ssthresh 时进入拥塞避免
- 每收到一个 ACK，cwnd 增加 **1/cwnd**（即每轮 RTT 增加 1）
- 增长速度是 **线性级**的：16 -> 17 -> 18 -> 19 ...
- 目的是**缓慢探测网络的极限容量**，避免突然把网络塞爆

### 慢开始 vs 拥塞避免的直观对比

```

    |
 24 |                                  拥塞避免阶段（线性增长）
    |                                 /
 20 |                               /
    |                             /
 16 |....慢开始门限(ssthresh)..../.......进入拥塞避免
    |                         /
 12 |                       /
    |                     /
  8 |                   /
    |                 /

    |             /
  2 |           /
    |         /
  1 |......./  慢开始阶段（指数增长）
    |

```

### 算法三：快重传（Fast Retransmit）

**场景：** 数据传输过程中，某个包丢了。

**传统做法（有问题的做法）：**

发送方等超时（RTO，超时重传计时器），超时了再重传。超时期间，发送方什么都不做，白白浪费时间。

**快重传的做法：**

接收方发现收到的包**不连续**（比如收到了 1、2、3，然后直接收到了 5），就知道 4 丢了。接收方**立即重复发送对 3 的 ACK**（重复确认，即 dup ACK），告诉发送方"我还在等 4，赶快发 4"。

- 当发送方收到 **3 个重复的 ACK**（即连续 4 次收到同样的 ACK 号）时，不用等超时，**立即重传丢失的报文段**
- 这大大减少了等待时间，提高了效率

**直觉类比：**

你给朋友寄了 1 号、2 号、3 号、4 号、5 号快递。朋友收到了 1、2、3、5，发现缺了 4 号，于是不停对你说"收到了 3 号，4 号呢？"说了 3 遍。你一听就明白 4 号丢了，**不等原定的等待时间**，马上补寄 4 号。

### 算法四：快恢复（Fast Recovery）

**跟快重传是配套的** ——快重传负责"发现丢包并重传"，快恢复负责"重传后怎么调整发送速度"。

**快恢复做的事：**

1. 收到 3 个重复 ACK 时，发送方知道**网络可能轻微拥堵**（还没到很严重的地步）
2. 把 ssthresh 降为当前 cwnd 的一半（保守起见）
3. 把 cwnd 也降为 ssthresh（即降一半），然后直接进入**拥塞避免**（线性增长）
4. 不再像传统做法那样把 cwnd 降到 1 重新慢开始

**为什么叫"快恢复"？**

- 相比传统做法（发生丢包就 cwnd=1，重新慢开始），快恢复**保留了较高的窗口值**
- 传统做法："出事了！窗口归零，从头开始！" -> 效率低
- 快恢复："出事了！窗口减半，从半路继续！" -> 效率高

### 四算法的完整配合流程

```
cwnd
  ^
  |  阶段1            阶段2                阶段3              阶段4
  |  慢开始           拥塞避免             丢包事件            丢包后
  |  (指数增长)        (线性增长)           (快重传+快恢复)     (拥塞避免)
  |
24 |                  ssthresh
  |                  ...... *
20 |                        / \
  |                       /   \
16 |......................*.....*...........ssthresh 减半
  |                     /         \              / \
12 |                   /           \            /   \
  |                  /             \          /     \
 8 |                *               \        /       \
  |               / \                \      /         \
 4 |             /   \                \    /           \
  |            /     \                \  /             \
 1 |***********        \               \/               \
  |                      \                              \
  +------------------------------------------------------------> 时间
```

**完整故事线（快递公司视角）：**

1. **慢开始**：新公司开张，从 1 辆车开始试探路况。每次成功往返，车辆翻倍。1->2->4->8...
2. **拥塞避免**：到了 16 辆车（ssthresh），路况摸清了。改为每次只增加 1 辆，小心试探网络极限。
3. **拥塞发生（丢包）**：当车辆达到 24 辆时，堵车了！一个包裹丢了。
   - 接收方连续 3 次说"少了这个包裹"（3 个重复 ACK）
   - 发送方不等超时，**立即重传**（快重传）
   - 同时把 ssthresh 降到 12（24 的一半），cwnd 也降到 12（快恢复）
4. **回到拥塞避免**：从 12 继续线性增长，再次逐渐试探网络极限。

## 四、总结对比表

| 概念 | 一句话 | 为什么是这个次数 |
|------|-------|----------------|
| 三次握手 | 双方确认"你在线、我也在线" | 两次不安全（旧请求可能浪费资源），四次浪费 |
| 四次挥手 | 双方分别说"我讲完了" | 全双工，你说完不代表对方说完，不能合并 |
| 慢开始 | 从 1 开始指数增长试探网络 | 避免一上来就塞爆未知的网络 |
| 拥塞避免 | 达到门限后线性增长 | 指数增长太猛会撞墙，改为小心试探 |
| 快重传 | 收到 3 个重复 ACK 就立即重传 | 不等超时，节省时间 |
| 快恢复 | 丢包后窗口减半而非归零 | 保留一定速度，避免从头开始太慢 |

### 记忆口诀

```

四次挥手：你拜拜->我确认->我拜拜->你确认
慢开始：   从 1 翻倍到门限
拥塞避免： 加 1 线性往上探
快重传：   三个重复立即传
快恢复：   窗口减半继续干
```
