## 色彩体系

第3章 · 原型页面生成
用HTML+CSS+Mock数据快速构建可交互原型，用于需求确认
🤔 思考一下
为什么原型阶段必须使用HTML静态页面，而不是直接用Vue/React框架？
A. Vue/React代码无法在浏览器直接运行
B. 原型阶段的目标是快速验证需求和交互流程，纯HTML无需构建工具链、无依赖安装、双击即可打开浏览，适合非技术人员参与评审
C. HTML不能实现复杂交互
D. 原型代码必须复用为生产代码

✅ 正确答案：B. 纯HTML无需构建工具链、无依赖安装、双击即可打开浏览

原型阶段的核心目标是「快速验证」，不是「工程化交付」。纯HTML+CSS+JS文件零依赖、零构建、零配置，产品经理/需求方/客户可以直接通过浏览器打开查看，无需安装Node.js或启动开发服务器。原型页面中的交互（弹窗/切换/表单填写）使用原生JS实现，Mock数据直接嵌入或通过JSON文件加载。原型验收通过后，原型代码即完成使命，不会进入生产。

第4章 · Mock数据与前端开发
使用Mock数据驱动前端开发，无后端也能完整运行
💬 思考与讨论
前端开发阶段，后端接口尚未就绪。前端工程师智能体应该如何设计Mock数据方案，使得从Mock到真实API的切换尽可能平滑？

推荐策略：①采用接口模拟层（Request Layer）设计——将所有API调用封装在一个独立的service模块中，Mock数据和真实API共用一个接口签名（相同的入参/出参类型）。②开发时通过配置开关切换数据源：`const USE_MOCK = true`，切换为false即对接真实API。③Mock数据使用Mock.js生成随机但符合Schema的仿真数据，非硬编码。④Mock数据的格式、字段类型必须与后端定义的接口文档（OpenAPI/Swagger）完全一致。⑤切换到真实API时只需修改配置文件，无需改动业务组件代码。

Mock数据与前端开发

Mock数据是前端独立开发的基石。前端工程师智能体根据接口文档自动生成Mock数据层，使前端项目在无后端服务的情况下也能完整运行和调试。

接口模拟层（Service Layer）

将API请求封装在service模块中，Mock和真实API使用相同的函数签名和返回类型，通过配置一键切换

智能Mock数据生成

AI根据接口Schema自动生成符合数据类型的仿真数据，支持嵌套对象、数组、枚举值和边界值覆盖

动态响应模拟

模拟网络延迟、错误响应、分页数据、Token过期等真实场景，提前处理前端异常逻辑

CRUD全流程模拟

Mock数据支持增删改查操作（内存操作），前端可以完整演示数据的新增、编辑、删除和查询

渐进式切换策略

支持按接口粒度切换到真实API——已完成的接口对接真实后端，未完成的继续使用Mock

数据状态保持

Mock数据存储在内存中，页面刷新后可重置或保持状态，支持多页面交互数据传递

AI生成的Mock Service层（Vue项目示例：Mock ↔ 真实API一键切换）
// ============================================
// service/product.js — 接口模拟层
// 同一套接口签名，支持 Mock / 真实API 一键切换
// ============================================

import request from '@/utils/request';

// 开发开关：true=使用Mock数据，false=对接真实API
const USE_MOCK = true;

// ---------- Mock 数据处理 ----------
const mockProducts = [
  { id: 1, name: '苹果手机', price: 8999, stock: 128,
    status: '上架', createdAt: '2024-01-15' },
  { id: 2, name: '机械键盘', price: 599,  stock: 56,
    status: '上架', createdAt: '2024-02-20' },
];

// 模拟网络延迟
const delay = (ms = 300) => new Promise(r => setTimeout(r, ms));

// ---------- 接口：商品列表 ----------
export async function getProductList(params) {
  if (USE_MOCK) {
    await delay(500);
    // 模拟分页和搜索过滤
    let filtered = [...mockProducts];
    if (params?.keyword) {
      filtered = filtered.filter(p =>
        p.name.includes(params.keyword)
      );
    }
    return {
      code: 200,
      data: { list: filtered, total: filtered.length }
    };
  }
  // 真实API调用（签名完全一致）
  return request.get('/api/products', { params });
}

// ---------- 接口：新增商品 ----------
export async function createProduct(data) {
  if (USE_MOCK) {
    await delay(300);
    const newProduct = { ...data, id: Date.now(), createdAt: new Date() };
    mockProducts.unshift(newProduct);
    return { code: 200, data: newProduct, message: '创建成功' };
  }
  return request.post('/api/products', data);
}

// ---------- 接口：删除商品 ----------
export async function deleteProduct(id) {
  if (USE_MOCK) {
    await delay(200);
    const idx = mockProducts.findIndex(p => p.id === id);
    if (idx > -1) mockProducts.splice(idx, 1);
    return { code: 200, message: '删除成功' };
  }
  return request.delete(`/api/products/${id}`);
}

// ---------- 切换为真实API：修改 USE_MOCK = false ----------
🚀Mock驱动开发最佳实践

①定义统一的API响应格式 `{ code, data, message }`，Mock与真实API保持一致 ②使用Mock.js生成随机但格式正确的数据，避免硬编码导致后期接口变化时大量修改 ③Mock数据覆盖正常响应、空数据、错误响应、超时四种场景 ④前端开发时以Mock数据为主，联调时将开关逐接口切换到真实API ⑤Mock Service层与业务组件解耦，切换数据源不影响UI代码

第5章 · 组件化开发与工程化
构建可维护、可复用的前端工程化项目
💬 思考与讨论
原型阶段生成的是独立的HTML静态页面，进入开发阶段后为什么要转为Vue/React组件化架构？直接基于原型代码开发不行吗？

答案：不行的原因如下：

①原型代码是单体HTML，所有样式和逻辑都在一个文件中，无法支撑大型应用的维护需求。②Vue/React提供组件化机制，将UI拆分为独立可复用的组件（如ProductCard、SearchBar、Pagination），修改一处影响全局。③工程化工具链（Vite/Webpack）提供模块热更新（HMR）、代码分割（Code Splitting）、Tree Shaking等功能，大幅提升开发体验和运行性能。④状态管理（Pinia/Vuex/Redux）解决跨组件数据共享问题，原型阶段完全不需要。⑤可维护性差异巨大：原型是「一次性验证工具」，工程化项目是「长期迭代的产品」。

正确的做法：原型只作为需求确认的参照物，开发阶段根据设计规范从头构建组件化工程。

组件化开发与工程化

工程化是生产级前端的基石。前端工程师智能体从零搭建Vue/React项目，按照组件化、模块化、规范化的原则组织代码，确保项目可维护、可测试、可扩展。

1
项目脚手架

使用Vite/Create React App初始化项目，配置TypeScript、ESLint、Prettier、路径别名

2
组件树设计

按页面区域拆分组件：布局组件（Header/Sidebar/Footer）、业务组件（ProductCard/OrderTable）、通用组件（Button/Modal/Pagination）

3
状态管理

设计全局状态（用户信息/购物车/主题）与局部状态（表单数据/弹窗显隐）的分层管理方案

// ============================================
// utils/request.js — 统一请求封装
// 自动处理跨域、鉴权、Token刷新、错误拦截
// ============================================

import axios from 'axios';
import { ElMessage } from 'element-plus';

const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || '/api',
  timeout: 15000,
  headers: { 'Content-Type': 'application/json' }
});

// 请求拦截器：自动携带Token
service.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('access_token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// 响应拦截器：统一错误处理 + Token自动刷新
service.interceptors.response.use(
  (response) => {
    const { code, data, message } = response.data;
    if (code === 200) return data;
    if (code === 401) {
      // Token过期 — 尝试刷新
      return refreshToken().then(() => {
        return service(response.config); // 重试原请求
      }).catch(() => {
        localStorage.clear();
        window.location.href = '/login';
      });
    }
    ElMessage.error(message || '请求失败');
    return Promise.reject(new Error(message));
  },
  (error) => {
    if (error.code === 'ECONNABORTED') {
      ElMessage.error('请求超时，请检查网络');
    } else if (!error.response) {
      ElMessage.error('网络连接失败');
    }
    return Promise.reject(error);
  }
);

export default service;
